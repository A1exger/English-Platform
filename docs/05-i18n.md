# 05 — Мультиязычность (i18n) и RTL

## 1. Поддерживаемые языки (этап 1)
`en` (основной/fallback), `ru`, `de`, `fr`, `nl`, `ar` (RTL).

Архитектура рассчитана на добавление новых языков **без изменения бизнес-логики** —
достаточно добавить файл ресурсов и зарегистрировать локаль.

## 2. Ключевой бизнес-инвариант
**Язык — атрибут пользователя (`user.locale`), а не урока или сессии.** Один
репетитор одновременно работает с учениками из Германии, Франции, Туниса,
Нидерландов, России и UK; каждый видит интерфейс на своём языке. Это **не влияет**
на совместный урок: видео и доска language-agnostic, общим является только контент
(холст, чат-сообщения), а вся «обвязка» (меню, кнопки, подсказки) — у каждого своя.

## 3. Frontend (Next.js + next-intl)
- Маршрутизация с локалью: `/[locale]/...` (`/de/dashboard`, `/ar/lessons`).
- **Автоопределение при первом входе**: `Accept-Language` → ближайшая поддерживаемая
  локаль; затем выбор сохраняется (cookie + `user.locale` в БД).
- **Быстрое переключение без перезагрузки**: смена локали меняет сегмент маршрута и
  ресурсы через next-intl; состояние приложения сохраняется.
- Форматирование дат, чисел, валют, относительного времени — через `Intl`
  (next-intl `useFormatter`), по активной локали и часовому поясу пользователя.
- Ресурсы — JSON по неймспейсам: `messages/{locale}/common.json`, `auth.json`,
  `lesson.json`, `billing.json`, … Тексты **не хардкодятся** в компонентах.

### RTL для арабского
- `dir="rtl"` на `<html>` при `locale === 'ar'`; логические CSS-свойства
  (`margin-inline-start`, `padding-inline-end`) вместо left/right.
- Зеркалирование направленных иконок (стрелки «назад/вперёд»), позиционирования
  сайдбара и выравнивания.
- Шрифт с поддержкой арабского (Noto Sans Arabic).

## 4. Backend (NestJS + nestjs-i18n)
- Резолвер локали: заголовок `Accept-Language` / query `?lang=` / `user.locale`.
- Локализованные ответы об ошибках и системные сообщения (`auth.invalid_credentials`
  и т.п.).
- Хранение `locale` пользователя в БД — источник истины для офлайн-каналов.

## 5. Уведомления (email, Telegram, чеки)
- При постановке уведомления в очередь фиксируется `notification.locale` = `user.locale`.
- Шаблоны писем/Telegram/счетов — по локали; чек/инвойс (PDF) формируется на языке
  пользователя.
- Воркер рассылки рендерит шаблон в зафиксированной локали (язык не «уплывёт», даже
  если пользователь сменит его позже).

## 6. Учебный контент vs интерфейс
- **Интерфейс** переводится автоматически по локали пользователя.
- **Учебные материалы**, созданные репетитором (доска, PDF, упражнения), остаются на
  языке оригинала и **не переводятся** автоматически.
- (Опционально) AI-перевод — только для подсказок, инструкций и системных сообщений,
  не для учебного контента.

## 7. Процесс локализации (ops)
- Источник правды — JSON-ресурсы в репозитории (на этапе роста — подключаемая
  TMS/CMS: Crowdin/Locize) для работы переводчиков без релизов кода.
- CI-проверка: отсутствие недостающих ключей относительно `en`; запрет «сырых»
  строк в JSX (ESLint-правило).
- Псевдолокализация в dev для отлова незавёрнутых строк и переполнения вёрстки.

## 8. Реализация в этом репозитории
- `apps/web` — каркас next-intl: конфиг локалей, middleware автоопределения,
  пример ресурсов для 6 языков, переключатель языка, RTL-обработка.
- `apps/api` — nestjs-i18n с ресурсами `en/ru/de/fr/nl/ar` и резолвингом локали.
