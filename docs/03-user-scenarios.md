# 03 — Пользовательские сценарии

Роли: **Репетитор (T)**, **Ученик (S)**, **Родитель (P)**, **Админ платформы (A)**.

## 1. Карта ролей и прав (RBAC)

| Возможность | T | S | P | A |
|---|:--:|:--:|:--:|:--:|
| Управление своими учениками (CRM) | ✓ | — | — | ✓ |
| Создание/перенос/отмена уроков | ✓ | частично¹ | — | ✓ |
| Бронирование свободных слотов | — | ✓ | ✓² | — |
| Проведение урока (видео+доска) | ✓ | ✓ | — | — |
| Выдача/проверка ДЗ | ✓ | submit | view | — |
| Настройка тарифов/пакетов | ✓ | — | — | ✓ |
| Оплата/пополнение баланса | — | ✓ | ✓ | — |
| Просмотр финансовой отчётности | свою | свою | ребёнка | вся |
| Управление платформой/модерация | — | — | — | ✓ |

¹ Ученик может отменить/перенести своё занятие в рамках политики репетитора.
² Родитель действует от имени привязанного ребёнка.

## 2. Сквозные сценарии (end-to-end)

### Сценарий A. Онбординг репетитора
1. Регистрация (email или Google/Microsoft) → выбор роли «репетитор».
2. Автоопределение языка интерфейса по браузеру; подтверждение в профиле.
3. Заполнение профиля: имя, часовой пояс, ставка/валюта, описание.
4. Настройка рабочего расписания (доступные слоты), тарифов и пакетов занятий.
5. Приглашение ученика по email/ссылке → создаётся `tutor_student`.
**Итог:** кабинет готов, можно планировать уроки.

### Сценарий B. Онбординг ученика (и родителя)
1. Переход по ссылке-приглашению → регистрация.
2. Язык интерфейса определяется автоматически (например, `ar` для ученика из Туниса)
   и может быть изменён — **независимо от языка репетитора**.
3. Заполнение профиля: уровень (CEFR), цели обучения, родной язык.
4. (Опц.) Родитель привязывается к профилю ребёнка (`parent_user_id`) и видит
   расписание, прогресс и платежи.
**Итог:** ученик видит предстоящие уроки и может бронировать/оплачивать.

### Сценарий C. Планирование и напоминания об уроке
1. T создаёт урок (индивидуальный/групповой/пробный): дата, длительность, ученик(и),
   цена; для серии — правило повторения (RRULE).
2. Система генерирует ссылку на занятие (комната LiveKit) и создаёт доску.
3. Ставятся задачи в очередь напоминаний: email + Telegram за 24 ч и за 1 ч,
   **на языке каждого получателя**.
4. Ученик видит урок в своём кабинете; может перенести/отменить по политике.

### Сценарий D. Бронирование учеником свободного слота
1. S открывает календарь репетитора со свободными слотами (в своём часовом поясе).
2. Выбирает слот → проверка баланса/пакета.
3. Если есть оплаченные занятия в пакете — бронь списывает 1 занятие (hold);
   иначе — переход к оплате.
4. Создаётся `lesson` + `lesson_participant`; уведомления обеим сторонам.

### Сценарий E. Проведение урока (ядро продукта)
1. За N минут до начала в кабинете активируется кнопка «Войти в урок».
2. `POST /lessons/:id/join` → API выдаёт токены LiveKit (видео) + доски + чата.
3. Единый экран урока: видео (HD, демонстрация экрана, шумоподавление, запись),
   интерактивная доска (рисование, текст, стикеры, изображения, PDF, перетаскивание,
   шаблоны, бесконечный холст), чат.
4. Совместная правка доски в реальном времени (Yjs CRDT); каждый видит интерфейс на
   своём языке, но контент доски общий.
5. Отмечается посещаемость (`attendance`): joined/left.
6. По завершении: статус `completed` → автосписание (баланс/пакет), сохранение
   снапшота доски и (опц.) ссылки на запись.

### Сценарий F. Домашнее задание
1. T выдаёт ДЗ (привязка к уроку, срок, материалы) → уведомление ученику.
2. S выполняет: текст/файлы/встроенные интерактивные упражнения/автотесты.
3. Автопроверяемые тесты оцениваются автоматически; письменные — вручную или с
   помощью AI-ассистента (предварительная проверка грамматики и фидбэк).
4. T оставляет оценку и комментарии → `progress_record` обновляется.

### Сценарий G. Оплата и баланс
1. S/P пополняет баланс или покупает пакет (Stripe/PayPal).
2. Вебхук провайдера → `transaction.status = succeeded` → `ledger_entry` (+),
   обновление `balance_cents` / `student_package`.
3. За проведённый урок — автосписание (см. E.6).
4. Генерируются счёт/чек (PDF) на языке пользователя; история транзакций доступна.

### Сценарий H. CRM и аналитика репетитора
1. В карточке ученика: профиль, история занятий, посещаемость, ДЗ, заметки, прогресс.
2. Дашборд аналитики: доход, число проведённых уроков, активность учеников,
   посещаемость, конверсия пробных → постоянных.

### Сценарий I. Смена языка интерфейса
1. Пользователь меняет язык в настройках (или быстрым переключателем в шапке).
2. UI мгновенно перерисовывается без перезагрузки (next-intl), выбор сохраняется в
   профиле (`user.locale`); последующие email/Telegram идут на новом языке; для `ar`
   включается RTL-раскладка.

## 3. Граничные случаи и политики
- **Отмена/перенос**: окно бесплатной отмены задаётся репетитором; поздняя отмена →
  списание по политике. Перенос ищет свободный слот, сохраняя оплату.
- **No-show**: статус `no_show`, списание по политике.
- **Разрыв связи на уроке**: переподключение к той же комнате/доске без потери
  состояния (CRDT-снапшот + LiveKit reconnection).
- **Групповой урок**: до 20 участников; бронь ограничена вместимостью; посещаемость
  по каждому.
- **Возврат средств**: `refund` через провайдера → компенсирующая `ledger_entry`.
